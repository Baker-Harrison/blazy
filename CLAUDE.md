# Commenting policy

The user (project owner) is not a professional developer. Always write extensive, plain-language comments throughout this codebase so the code is easy for a non-developer to read and follow.

- Comment every file, every function/component, and every non-trivial block — not just the "why," but also the "what," in everyday language (avoid jargon, or define it inline when unavoidable).
- Explain things the way you'd explain them to someone who doesn't code: use analogies and plain descriptions instead of assuming familiarity with programming terms.
- Prefer more comments over fewer. When in doubt, add an explanation.
- This overrides the general "don't over-comment" convention — for this project specifically, verbose, beginner-friendly comments are the desired style. Keep this style for all future edits to this codebase, not just existing files.
